2 · Critique

Grouped by lens. Each item carries a severity tag — high blocks the job, med creates friction, low is polish.

Usability does the screen let you do the job?

  • high Mobile rows show only the title. Priority, category, due-date, status are all desktop-only (lg:contents). When grouping by date you can't see priority — and vice versa — so you can't actually triage on mobile.
  • high Status & priority can't be changed inline. The pop-overs are desktop-only; the only mobile path is open the editor → tap tab → save → close. 4 taps for what should be 1.
  • high No swipe / long-press actions. Complete and delete are not reachable from the list — you have to enter the editor.
  • med "New task" is a tab, not an action. The bottom nav has only 2 entries (home / new task) and the "new task" path is a full screen instead of a quick-add sheet — heavy for the most-used action.
  • med No search, no filter, no sort. Once the list passes ~30 items the only tool is collapse-the-group; you can't find a known task quickly.
  • low Edit pencil overlaps the row tap target. Whole row should open the editor; pencil is redundant icon noise.

UX & information architecture does it match how the user thinks?

  • high No "what matters now" surface. The first thing you see is a 4-way group toggle, not "you have 2 overdue, 4 due today". A task app should answer the standing question first.
  • med Editor and creator diverge. Editor has Status; creator doesn't. Editor has tags, source_type, source_ref; creator doesn't. New tasks always start as "active" with no way to set status — surprising.
  • med Save lives at the bottom of a long scroll. On a phone with software keyboard you must scroll past Status / Priority / Meta to commit. A sticky save in the header would cut the loop.
  • med "Done" tasks just sort to the bottom. They still occupy the same group, padding the list. Hide-by-default with a "show 3 completed" toggle is more honest.
  • low "Sometime" bucket is hidden by sort. Tasks without a date never appear before "Today" so users assume they're lost.
  • low Group toggle resets to "by date" via URL param. User preference doesn't persist between sessions.

Visual design does it look intentional?

  • med Italic-bold placeholder. "Add a new task..." in italic + 26px bold reads like a heading the user accidentally bolded; a plain regular weight placeholder is calmer.
  • med Hairlines do all the work. Section headers, rows, tabs are separated only by 1-px lines. With nothing else to anchor the eye, the screen feels like a long ledger.
  • med Native date input + native <select>. Inconsistent typography vs. the rest of the app and unstyled in dark mode (the date picker icon is filtered, but the chevrons differ).
  • low One accent color carries every "active" state. Group toggle, write/preview, status, priority, primary button — all emerald. Hierarchy collapses; nothing reads as "do this next".
  • low Status colors only appear in dropdowns (mobile-hidden). The carefully-configured status / priority palettes are invisible on the most-used surface.
  • low Bottom nav has 2 items in a 2-col grid. On a 360 px screen those buttons are ~170 px wide, far past the comfortable thumb target — they look unintentional, like "we'll add more later".

Accessibility & touch WCAG / Apple HIG basics

  • med 14-px section labels at 0.08em uppercase with the lowest-contrast token (#71717a on #14141a → 4.6:1). Just above the AA floor; one render tweak away from failing.
  • med Edit pencil is 32×32 px. Below Apple's 44 pt minimum and Google's 48 dp minimum. Easy to mis-tap when adjacent to the checkbox.
  • low Bottom nav covers content. Padding-bottom is 28px (`pb-28`) but inline forms (e.g. New Task) extend beneath it; on small screens the Save button can sit under the nav.
  • low No haptic / no completed-state toast. Tapping a checkbox is silent; the only feedback is a tiny "Updating..." pill in the corner.

5 · Gesture proposal — swipe & long-press

Resolves the high "no swipe / long-press actions" critique. Same task card; three interaction states. Swipe-right completes (low-risk, frequent); swipe-left deletes (destructive, requires past-threshold or undo); long-press opens a context sheet for everything else (status, priority, category, edit).

GestureOutcomeBehavior & safeguards
Swipe rightComplete (or un-complete)Reveal at 12% · commit at 25% of card width · light haptic at threshold · row collapses with a strikethrough crossfade · undo via long-press menu.
Swipe leftDeleteReveal at 12% · button visible at 25–60% · commit only past 60% (or by tapping the revealed Delete button) · medium haptic at 60% · 5-second Undo toast guards the destructive action.
Long-pressContext sheet (all actions)~400ms hold · card lifts (scale 1.02) with shadow + emerald glow + medium haptic · half-sheet covers ~55% of screen · all 7 task actions accessible · drag-down or tap-out to dismiss.
TapOpen editor (full screen)Unchanged from today — the row tap target is the whole card minus the checkbox.
Tap checkboxToggle completeKept as an explicit, accessible alternative to the swipe gesture (matches `role="checkbox"` semantics already in the code).
