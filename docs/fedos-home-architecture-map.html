The operating model

FedOS Home now contains the intelligence capability. Memory remains separate because it is the canonical context source, not because it is a second runtime.

FedOS Memory

Canonical

The stable operating memory: personal context, priorities, preferences, stakeholders, principles, and approved digest material.

  • 1Owns source Memory files and long-lived context.
  • 2Produces or stores an approved digest used as prompt context.
  • 3Receives only reviewable update suggestions, never silent writes.
  • 4Stays separate from Home so the source of truth remains inspectable.

FedOS Home

Integrated runtime

The user-facing command center and intelligence runtime. Home ingests signals, reasons over context, stores briefings, and turns approved proposals into tasks.

  • 1Fetches Outlook signals and normalizes them into `BriefingSignal` records.
  • 2Runs the intelligence layer: hygiene, compaction, prompt assembly, LLM reasoning, and validation.
  • 3Persists `BriefingPackage` and `ProposedAction` records in PostgreSQL.
  • 4Owns approve, reject, defer, task creation, Home UI, APIs, and the Debug Console.

End-to-end operating flow

The product loop starts with fresh signals and Memory context, then ends in reviewable decisions inside Home. Proposed actions are not tasks until approved.

Pipeline logic

Outlook fetch

Recent mail and upcoming calendar signals, with token status and warnings.

Normalize

Graph payloads become compact `BriefingSignal` records.

Hygiene

Obvious machine noise removed; duplicate source IDs collapsed.

Compact

Threads and repeated sources are summarized without losing traceability.

Memory

Approved digest is inserted as stable, read-only context.

LLM

Judgment-heavy synthesis returns narrative, priorities, recommendations, proposed actions, uncertainty.

Validate

Zod-style schema validation keeps malformed output out of the product path.

Persist or preview

Product runs write records; debug dry-runs inspect the same pipeline without DB writes.

The daily user loop

The user experience should stay calm: brief first, decision second, task execution third. The system assists judgment but does not silently act.

Decision semantics

Approve Creates one durable Home task and links it back to the originating proposal.
Reject Marks the proposal as not useful without creating task debt.
Defer Keeps the proposal visible as not-right-now rather than throwing it away.
Challenge next Future HCI-09 path: ask why, correct the briefing, and preserve a revision.
